1.  Background
The Risk and Product teams require a centralized, live view of key bonus-related metrics across operators and campaigns. Current analysis is fragmented and retrospective, limiting strategic and operational decision-making.


Monitoring performance and behavior related to bonus assignments is critical to financial and operational control. Without live KPIs, it is difficult to spot anomalies in time to react. Common examples include:


* High redemption ratios across specific operator segments
* Underperforming campaigns that fail to drive intended player engagement
* Abuser concentration patterns that expose the platform to financial risk


Strategic Context
This PoC evaluates RisingWave as a real-time streaming engine to power the Customer Attributes Platform — a foundational component that enables live, customer-level metric delivery across internal Risk and Product stakeholders.
	

2.  Objective
Evaluate RisingWave as a real-time engine partner by assessing its capability to:


* Process streaming data with subsecond latency
* Handle stateful computations at scale
* Deliver materialized views for customer-level metrics
* Integrate with the existing Kafka-based data infrastructure


3.  Scope
The PoC is divided into two sequential phases: a Requirements Phase that validates core streaming capabilities, and a Verification Phase that stress-tests resilience and operational properties.


3.1  Requirements Phase
The following requirements must be satisfied for RisingWave to be considered a viable candidate. Each item is independently testable and maps directly to a test case or benchmark.


ID
	Requirement[c][d]
	Description
	R1
	Catch-up
	Ability to process accumulated backlog when starting or resuming a pipeline. Validates readiness after planned downtime or cold-start scenarios.
	R2
	Backfill
	Support for reprocessing historical data to recalculate metrics. Enables retroactive corrections without a full pipeline rebuild.
	R3
	Late-Arriving Data
	Handle out-of-order events within acceptable time windows. Ensures aggregation correctness despite network delays or producer lag.
	R4
	Subsecond Latency
	p95 end-to-end latency from Kafka source to Kafka sink must remain below 1 second. Non-negotiable for live bonus monitoring and real-time alerting.
	

3.2  Verification Phase
Verification items go beyond basic functionality to validate system behaviour under adversarial or degraded conditions. All three verifications must pass for the PoC to be considered successful.


ID
	Requirement
	Description
	V1
	State Failure Recovery
	Recover pipeline state from checkpoints without data loss. Validates RisingWave's checkpoint mechanism and state-backend durability under failure.
	V2
	Data Spikes
	Handle burst traffic[e] spikes without data loss or significant latency degradation. Simulates high-volume bonus event bursts typical of promotional campaigns.
	V3
	Failure Recovery
	Full pipeline recovery from any failure within some seconds. Tests end-to-end resilience across source, processing layer, and sink.

4.  Test Cases
Two use cases are defined for this PoC. They are representative of the broader Customer Attributes Platform workload and cover both windowed aggregation (UC1) and ratio computation (UC2).


4.1  Use Case 1: Casino Real Bet Amount


UC1  Casino Real Bet Amount
	Business Definition  
Total amount of real money bets placed by the customer in the casino over the past two weeks. Used to assess recent customer activity, identify trends, and design targeted engagement or retention strategies.
Technical Implementation
1. Transform source data from Kafka topic(s)
2. Calculate rolling 2-week sum of real bet amounts per customer
3. Emit updates to destination Kafka topic in near-real-time
	

4.2  Use Case 2: Casino Turnover Percentage


UC2  Casino Turnover Percentage
	Business Definition  
Ratio of casino betting turnover to total betting turnover. Used to assess a customer's preference for casino activity relative to their overall betting behavior.
Technical Implementation
1. Transform source data from Kafka topic(s)
2. Calculate ratio of casino turnover to total turnover per customer
3. Emit updates to destination Kafka topic in near-real-time

4.3  Use Case 3: Sportsbook Live Trends Engine and Betslip Recommendation Engine


UC3  Casino Turnover Percentage
	Business Definition  
An engine that digest single selections, while also combinations of selections (BetBuilders, accumulators)  with related entity information (sport-league-zone-event-player(if applicable)-live) and is able to produce: 
* Social Proof (times leg was placed- selection/ bb)
* Top features per entity (top players/top betbuilders,/top betbuilder of top players/ most trending accumulators e.t.c)
* A live Betslip recommendation (Live Betslip Co-occurrence & Affinity Engine (X+1)) engine:
        Leverage live building betslip data to deliver session-based next-item recommendations(basically upselling), dynamically suggesting the most trending next selection combinations (Betbuilders or accumulators) to add to an active betslip with their “social proof”.  
Technical Implementation
   1. Input is currently writes of selection/matchcombo/ accumulators deltas in timeframed buckets (eg. Selection X +5k placement in timeframe Z)
   2. Querying is based on a total timeframe aggregation related to feature and use-case (not fixed) and can stem from 5-10 minutes up to 48 hours.


Workload Profile & Data Characteristics 

   1. Writes rate (adjustable): approx max: 1400/sec
   2. Query rate (Top features): approx max (currently) : 70/sec (expected): 150-200/sec
   3. Query rate (Betslip Recommendation): estimate: 200-250/sec.
   4. Combinations cardinality: selections: 55.000, bbs: 70.000 * times 70 timeframe buckets

System Performance Targets

      1. Ingestion Latency (Freshness): acceptable max under 1 min max delay. (unless backfill).
      2. Acceptable response time goals: 
Top features(1-2 sec), 
social proof (1-2 sec), 
Betslip Recommendation (3-5  sec) * (since not offered yet, this is an approx. based on clickhouse projected performance)

The PoC will be considered successful if all four of the following criteria are met. Partial success on any criterion does not constitute overall pass.

Appendix A — Glossary


RisingWave
	An open-source, cloud-native streaming database built on PostgreSQL-compatible SQL, designed for real-time analytics on streaming data.
	Kafka
	Apache Kafka — a distributed event-streaming platform used as the data source and sink in this PoC.
	Materialized View
	A pre-computed query result that is automatically updated as underlying data changes, enabling low-latency reads.
	p95 Latency
	The 95th-percentile end-to-end processing latency — i.e., 95% of events are processed within this duration.
	Backfill
	The process of reprocessing historical events to populate or recalculate metrics for past time windows.
	Catch-up
	The ability of a streaming pipeline to rapidly process a backlog of events that accumulated during a downtime period.
	Checkpoint
	A point-in-time snapshot of pipeline state used to resume processing without data loss after a failure.
	UC
	Use Case — a concrete business scenario implemented as a streaming pipeline for evaluation.
	CAP
	Customer Attributes Platform — the target system this PoC feeds into.
