1. BACKGROUND
-------------
The Risk and Product teams require a centralized, live view of key bonus-related
metrics across operators and campaigns. Current analysis is fragmented and
retrospective, limiting strategic and operational decision-making.

Monitoring performance and behavior related to bonus assignments is critical to
financial and operational control. Without live KPIs, it is difficult to spot
anomalies in time to react. Common examples include:
  - High redemption ratios across specific operator segments
  - Underperforming campaigns that fail to drive intended player engagement
  - Abuser concentration patterns that expose the platform to financial risk

Strategic Context: This PoC evaluates RisingWave as a real-time streaming engine
to power the Customer Attributes Platform — a foundational component that enables
live, customer-level metric delivery across internal Risk and Product stakeholders.


2. OBJECTIVE
------------
Evaluate RisingWave as a real-time engine partner by assessing its capability to:
  - Process streaming data with subsecond latency
  - Handle stateful computations at scale
  - Deliver materialized views for customer-level metrics
  - Integrate with the existing Kafka-based data infrastructure


3. SCOPE
--------
The PoC is divided into two sequential phases: a Requirements Phase that validates
core streaming capabilities, and a Verification Phase that stress-tests resilience
and operational properties.

3.1 Requirements Phase
  R1  Catch-up            Ability to process accumulated backlog when starting or
                          resuming a pipeline. Validates readiness after planned
                          downtime or cold-start scenarios.
  R2  Backfill            Support for reprocessing historical data to recalculate
                          metrics. Enables retroactive corrections without a full
                          pipeline rebuild.
  R3  Late-Arriving Data  Handle out-of-order events within acceptable time windows.
                          Ensures aggregation correctness despite network delays or
                          producer lag.
  R4  Subsecond Latency   p95 end-to-end latency from Kafka source to Kafka sink must
                          remain below 1 second. Non-negotiable for live bonus
                          monitoring and real-time alerting.

3.2 Verification Phase
  V1  State Failure Recovery  Recover pipeline state from checkpoints without data
                              loss. Validates RisingWave's checkpoint mechanism and
                              state-backend durability under failure.
  V2  Data Spikes             Handle burst traffic spikes without data loss or
                              significant latency degradation. Simulates high-volume
                              bonus event bursts typical of promotional campaigns.
  V3  Failure Recovery        Full pipeline recovery from any failure within some
                              seconds. Tests end-to-end resilience across source,
                              processing layer, and sink.


4. TEST CASES
-------------
Two use cases are defined for this PoC (UC1, UC2); a third (UC3) is documented for
context. They cover windowed aggregation (UC1) and ratio computation (UC2).

4.1 Use Case 1: Casino Real Bet Amount
  Business Definition: Total amount of real money bets placed by the customer in the
  casino over the past two weeks. Used to assess recent customer activity, identify
  trends, and design targeted engagement or retention strategies.
  Technical Implementation:
    1. Transform source data from Kafka topic(s)
    2. Calculate rolling 2-week sum of real bet amounts per customer
    3. Emit updates to destination Kafka topic in near-real-time

4.2 Use Case 2: Casino Turnover Percentage
  Business Definition: Ratio of casino betting turnover to total betting turnover.
  Used to assess a customer's preference for casino activity relative to their
  overall betting behavior.
  Technical Implementation:
    1. Transform source data from Kafka topic(s)
    2. Calculate ratio of casino turnover to total turnover per customer
    3. Emit updates to destination Kafka topic in near-real-time

4.3 Use Case 3: Sportsbook Live Trends Engine and Betslip Recommendation Engine
  Business Definition: An engine that digests single selections, plus combinations
  of selections (BetBuilders, accumulators) with related entity information
  (sport-league-zone-event-player(if applicable)-live), able to produce:
    - Social Proof (times a leg was placed — selection / betbuilder)
    - Top features per entity (top players / top betbuilders / top betbuilder of top
      players / most trending accumulators, etc.)
    - A live Betslip recommendation (Live Betslip Co-occurrence & Affinity Engine
      (X+1)): leverage live building-betslip data to deliver session-based next-item
      recommendations (upselling) — dynamically suggesting the most trending next
      selection combinations (BetBuilders or accumulators) to add to an active
      betslip, with their "social proof".
  Technical Implementation:
    1. Input is currently writes of selection/matchcombo/accumulator deltas in
       timeframed buckets (e.g. Selection X +5k placements in timeframe Z)
    2. Querying is a total-timeframe aggregation related to feature and use-case
       (not fixed) — from 5-10 minutes up to 48 hours.
  Workload Profile & Data Characteristics:
    1. Writes rate (adjustable): approx max 1400/sec
    2. Query rate (Top features): currently ~70/sec, expected 150-200/sec
    3. Query rate (Betslip Recommendation): estimate 200-250/sec
    4. Combinations cardinality: selections 55,000; bbs 70,000 × 70 timeframe buckets
  System Performance Targets:
    1. Ingestion latency (freshness): acceptable max under 1 min (unless backfill)
    2. Acceptable response times: Top features 1-2s, social proof 1-2s,
       Betslip Recommendation 3-5s (approx., based on ClickHouse projections)

1. OVERVIEW
-----------
Defines scope, objectives, deliverables, and timeline for a PoC evaluating
RisingWave as a potential addition to the Big Data Department's stack. Focuses on
enhancing real-time processing alongside the existing Kafka and Spark Streaming
infrastructure — covering interoperability with the wider tooling ecosystem, data
type/format support, complex stream handling, scaling economics, and serving-layer
latency.


2. BACKGROUND & MOTIVATION
--------------------------
Current stack: Apache Kafka (event streaming) + Apache Spark Streaming (scalable
real-time processing). Robust but operationally complex, with latency constraints
and limited native low-latency SQL stream analytics without heavy engineering.

RisingWave is a distributed SQL streaming database unifying stream ingestion,
incremental computation, and low-latency serving in one system: PostgreSQL-
compatible SQL, native Kafka integration, built-in state management, Apache Iceberg
support. May interoperate with Atlan, Databricks, dbt, Dagster, and more.


3. OBJECTIVES
-------------
  - Core streaming performance: end-to-end latency, throughput, MV freshness
  - Kafka integration without major architectural disruption
  - Interoperability: Atlan, Databricks, Iceberg, Power BI, Dagster, incident.io, dbt
  - Data type breadth & schema management: complex types, Schema Registry, Avro, Protobuf
  - Complex stream handling: multi-stream joins, late data, sessionization, out-of-order
  - Scaling model and its correlation to performance and infrastructure cost
  - Serving-layer latency from downstream consuming applications
  - Operational simplicity, fault tolerance, observability


4. SCOPE OF WORK
----------------
4.1 In Scope
  - Interoperability with Atlan, Databricks, Iceberg, Power BI, Dagster, incident.io,
    dbt, Prometheus and Grafana
  - Schema Registry integration (Confluent-compatible), Avro, Protobuf
  - Complex data types (nested JSON, arrays, maps, structs, UNIONs)
  - Complex streams: multi-stream joins, windowed aggregations, CDC, late/out-of-order
  - Scaling experiments: vertical & horizontal under synthetic load with cost tracking
  - Serving-layer assessment: latency via PG wire / REST / Kafka sink

4.2 Out of Scope
  - Full production deployment or live pipeline migration
  - Formal security audit / penetration testing
  - SLA definition for production
  - BI tool development beyond connectivity validation

8. DATA TYPES, FORMATS & SCHEMA REGISTRY
----------------------------------------
8.1 Complex Data Types: nested JSON (>=3 levels), arrays & array-of-structs, maps
   with dynamic keys, union/nullable, timestamp variants (epoch millis, ISO 8601,
   Avro logical types), large decimal/numeric financial fields.
8.2 Schema Registry: consume SR-managed topics without custom deserialization;
   schema evolution (backward-compatible add, removal, type widening); subject
   naming strategies (TopicNameStrategy, RecordNameStrategy); error handling for
   unregistered/incompatible versions.
8.3 Avro & Protobuf: end-to-end Avro ingestion into MVs; end-to-end Protobuf (proto3)
   incl. nested messages; correctness vs. a reference Spark consumer; perf delta
   JSON vs Avro vs Protobuf at equal throughput.


9. COMPLEX STREAM SCENARIOS
---------------------------
  Multi-Stream Joins      Join 3+ streams on shared keys, different rates/cardinalities
  Temporal/Window Joins   As-of joins between live stream and slowly-changing dimension
  Tumbling & Sliding      1-min tumbling and 5-min sliding aggregations under load
  Session Windows         Session aggregation with inactivity gaps (30s, 5min)
  Late & Out-of-Order     Inject up to 2-min-delayed events; watermark/late-data policy
  Backpressure            Spike to 3x throughput; observe queue depth, latency, recovery


10. SCALING MODEL, PERFORMANCE & COST
-------------------------------------
10.1 Disaggregated compute-storage: compute scales independently of object storage
   (S3 or compatible); nodes added/removed without downtime; state persisted to
   object storage (decouples compute cost from state size).
10.2 Scaling matrix (50k ev/s baseline):
   1x baseline   2 nodes   50k ev/s    latency, CPU, cost/hr
   2x            4 nodes   100k ev/s   linear scaling check
   4x            8 nodes   200k ev/s   efficiency-drop check
   Scale-down    2 nodes   50k ev/s    recovery time, latency
10.3 Cost: compute $/hr, object storage $/GB-month, cost-per-million-events per tier,
   TCO vs equivalent Spark Streaming cluster, identification of cost cliffs.


11. SERVING LAYER & APPLICATION LATENCY
---------------------------------------
11.1 RisingWave serves MV results via PostgreSQL wire protocol (any Postgres client),
   plus Kafka sinks and HTTP/REST connectors.
11.2 Scenarios: direct PG wire (point lookups + range scans, p50/p95/p99);
   Power BI DirectQuery refresh; Kafka sink source-to-sink lag; 20-50 concurrent reads.
11.3 Latency Targets:
   PostgreSQL wire (point lookup)   p50 < 10ms   p99 < 50ms    Critical
   PostgreSQL wire (range scan)     p50 < 50ms   p99 < 200ms   Critical
   Kafka sink (source-to-sink lag)  p50 < 500ms  p99 < 1s      High
   Power BI DirectQuery refresh     p50 < 3s     p99 < 10s     High
   Concurrent reads (50 conns)      p50 < 50ms   p99 < 300ms   High
